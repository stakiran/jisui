# watchdog のイベントキューに関する挙動メモ

## 現象

ocr_agent が処理中に、監視ディレクトリ内のまだ処理されていないファイルを手動で削除した。
ocr_agent がそのファイル番号に到達したとき、`No such file or directory` エラーが連続で発生した。

## 原因

### 1. watchdog のイベントキューはファイル作成時点で確定する

watchdog はファイルシステムの変更イベントをキューに溜める。
ocr_agent は1件ずつ同期で API を呼ぶため、処理中に後続のイベントがキューに蓄積される。
後からファイルを削除しても、既にキューに入ったイベントは消えない。

### 2. 1ファイルの保存で2つのイベントが発生する

PNG を保存すると、OS レベルでは「ファイル作成 → データ書き込み」の2段階になることが多い。

- `on_created` — ファイルが新規作成された瞬間
- `on_modified` — ファイルの内容が変更された瞬間

この2つは同じキューに入る（キューが2つあるわけではない）。

### 3. 正常時は重複処理されない

`_handle` 内で `self.processed` に処理済みファイル名を記録しているため、
`on_created` で処理成功 → `on_modified` が来ても `processed` にあるのでスキップされる。

### 4. エラー時に無限リトライになる原因

```python
except Exception as e:
    print(f"[ERROR] ...")
    self.processed.discard(path.name)  # ← これ
```

エラー時に `processed` からファイル名を外しているため、
次の `on_modified` イベントが来たときにスキップされず再実行される。
ファイルが存在しないのだから何度やっても同じエラーになる。

## まとめ

- watchdog のイベントキューは発生時点で確定し、後からファイルを消してもキューからは消えない
- 1ファイルにつき `on_created` + `on_modified` の2イベントが入りうる
- エラー時の `discard()` により、同じファイルで2回エラーが出る
- キューに入ったイベントをすべて処理し終えれば、それ以上は発生しない（待機状態に戻る）
